# Identity and Access: OAuth 2.0, OIDC, RBAC, PAM, Service Accounts

## The Big Picture

Identity and access management (IAM) is the enforcement point for the principle of least privilege. It answers two questions: who are you (authentication), and what are you allowed to do (authorization).

```
IDENTITY AND ACCESS LANDSCAPE
+-----------------------------------------------------------------------+
|                                                                       |
|  AUTHENTICATION              AUTHORIZATION                            |
|  (who are you?)              (what can you do?)                       |
|  +-----------------------+   +---------------------------+            |
|  | Password + MFA        |   | RBAC (Role-Based)         |           |
|  | OAuth 2.0 flows       |   | ABAC (Attribute-Based)    |           |
|  | OIDC identity token   |   | ReBAC (Relationship-Based)|           |
|  | SAML 2.0 (legacy)     |   | Zanzibar (Google)         |           |
|  | Certificate-based     |   +---------------------------+           |
|  | WebAuthn / FIDO2          |                                       |
|  +-----------------------+   PRIVILEGED ACCESS                       |
|                               +---------------------------+           |
|  SERVICE IDENTITY             | PAM (Privileged Access    |           |
|  +-----------------------+   | Management)               |            |
|  | Managed identities    |   | JIT (Just-in-Time)        |           |
|  | Service principals    |   | Session recording         |           |
|  | Workload identity fed.|   | Credential vaulting       |           |
|  | Client credentials    |   +---------------------------+           |
|  +-----------------------+                                           |
+-----------------------------------------------------------------------+
```

## OAuth 2.0: Authorization Framework

OAuth 2.0 (RFC 6749, 2012) is an authorization framework — it enables a client to obtain limited access to a resource on behalf of a resource owner, without sharing credentials.

### Core Roles

```
RESOURCE OWNER: The user (or another service) that owns the resource
CLIENT:         The application requesting access (web app, mobile app, service)
AUTHORIZATION SERVER: Issues tokens after authenticating resource owner
  (Azure AD / Entra ID, Okta, Auth0, AWS Cognito)
RESOURCE SERVER: The API/service that accepts tokens and enforces access
```

### OAuth 2.0 Flows

#### Authorization Code + PKCE (Preferred for Web and Mobile)

```
Browser / App                Auth Server              Resource Server
     |                           |                          |
     |-- Authorization Request ->|                          |
     |   (client_id, scope,      |                          |
     |    redirect_uri,          |                          |
     |    code_challenge,        |                          |
     |    response_type=code)    |                          |
     |                           |                          |
     |   User authenticates &    |                          |
     |   consents on Auth Server |                          |
     |                           |                          |
     |<-- Authorization Code ----|                          |
     |   (via redirect to        |                          |
     |    redirect_uri)          |                          |
     |                           |                          |
     |-- Token Request --------> |                          |
     |   (code, code_verifier,   |                          |
     |    client_id)             |                          |
     |                           |                          |
     |<-- Access Token + --------|                          |
     |   Refresh Token           |                          |
     |                           |                          |
     |-- API Request with ---------------------->           |
     |   Bearer Token                                       |
     |                                                      |
     |<-- API Response ------------------------------------- |

PKCE (Proof Key for Code Exchange, RFC 7636):
  code_verifier: random 32-byte value, generated by client
  code_challenge: SHA-256(code_verifier), sent in auth request
  code_verifier: sent in token request
  → Server validates: SHA-256(verifier) == challenge
  → Prevents authorization code interception attacks
  → Required for public clients (mobile, SPA); recommended for all
```

#### Client Credentials (Service-to-Service)

```
Service A                    Auth Server              Service B (API)
   |                              |                        |
   |-- Token Request -----------> |                        |
   |   (client_id, client_secret, |                        |
   |    or certificate assertion, |                        |
   |    grant_type=client_creds)  |                        |
   |                              |                        |
   |<-- Access Token ------------ |                        |
   |   (no user context,          |                        |
   |    audience=service-b)       |                        |
   |                              |                        |
   |-- API Call with Bearer Token ---------------------> |
   |                                                       |
PREFERRED: Use Azure Managed Identity instead of client_secret
  → No secret to store, rotate, or leak
  → Token obtained from Azure Instance Metadata Service
```

#### Device Flow (IoT / CLI / Headless)

```
Device / CLI              Auth Server            Browser (user)
    |                          |                      |
    |-- Device Auth Request -> |                      |
    |<-- device_code,          |                      |
    |   user_code,             |                      |
    |   verification_uri       |                      |
    |                          |                      |
    | Display: "Visit https://microsoft.com/devicelogin"
    | "Enter code: WDJB-MJHT" |                      |
    |                          |                      |
    | Poll for token every 5s  |   User visits URI,   |
    |                          |   enters user_code,  |
    |                          |   authenticates    > |
    |                          |                      |
    |<-- Access Token ---------|                      |
```

#### Implicit Flow (Deprecated)

The Implicit flow returned access tokens directly in the URL fragment (no code exchange). Deprecated because:
- Access tokens in URLs are logged in browser history, server logs, referer headers
- No refresh tokens
- No PKCE protection

**Do not implement Implicit flow for new applications.** Use Authorization Code + PKCE instead.

## OIDC: OpenID Connect (Identity on Top of OAuth 2.0)

OAuth 2.0 is an authorization framework — it grants access but does not define who the user is. OIDC (OpenID Connect, 2014) adds the identity layer.

```
OIDC ADDITIONS TO OAUTH 2.0:
  ID Token: JWT containing user identity claims
    {
      "iss": "https://login.microsoftonline.com/{tenant}/v2.0",
      "sub": "unique_user_identifier",
      "aud": "your_client_id",
      "exp": 1735689600,
      "name": "Jane Smith",
      "email": "jane@contoso.com",
      "oid": "azure-ad-object-id",
      "tid": "azure-ad-tenant-id"
    }

  UserInfo Endpoint: GET /userinfo with access token → user claims
  Discovery Endpoint: /.well-known/openid-configuration
    → Returns JWKS URI, authorization endpoint, token endpoint, etc.

OAUTH 2.0 vs. OIDC:
  OAuth 2.0:  "Can app X read my Google Drive?" (authorization)
  OIDC:       "Who is this user? Is it jane@contoso.com?" (authentication)
  In practice: use both together — OAuth 2.0 scopes for authorization,
               OIDC ID token for user identity
```

## RBAC, ABAC, and ReBAC

### RBAC: Role-Based Access Control

```
RBAC MODEL:
  User → Role → Permissions
  Jane → ReadOnlyAdmin → {Read: storage, Read: databases}
  Bob → Contributor → {Read: *, Write: *, No: delete, No: IAM}
  Alice → Owner → {Read: *, Write: *, Delete: *, Manage: IAM}

AZURE RBAC:
  Scope: management group → subscription → resource group → resource
  Role assignment: (identity, role, scope)
  Built-in roles: Owner, Contributor, Reader, + 100+ service-specific roles
  Custom roles: define specific permissions

LIMITATIONS OF RBAC:
  Role explosion: as permissions become more granular, role count explodes
  "Contributor on storage-account-X but not storage-account-Y" →
    requires either two roles or a custom role per resource
  → Use resource-scoped role assignments rather than creating a new role per case
```

### ABAC: Attribute-Based Access Control

```
ABAC MODEL:
  Access = f(subject attributes, resource attributes, environment attributes)
  "Allow read IF user.department == resource.ownerDept AND time.hour IN [9,17]"

ADVANTAGE: More expressive than RBAC
DISADVANTAGE: Policy complexity grows quickly; audit is harder

AZURE ABAC (preview):
  Storage blob ABAC conditions: allow read IF blob tag "project" == user attribute "project"
  → More granular than subscription/RG scope RBAC
```

### ReBAC: Relationship-Based Access Control

```
ReBAC MODEL: Access based on graph relationships
  User A is a member of Group G
  G has access to Document D
  → A has access to D

GOOGLE ZANZIBAR (2019):
  Powers Google Docs sharing, YouTube, Drive sharing
  "User can view document if User is viewer of document OR
   User is member of group that is viewer of document OR
   User is editor..."
  Stores tuples: (object, relation, user)
  Evaluates: Check(user, relation, object) → allow/deny

OPEN SOURCE: SpiceDB (AuthZed), OpenFGA, Ory Keto
USE CASE: Anything with hierarchical or relationship-based sharing
  (document sharing, team memberships, organizational hierarchies)
```

## PAM: Privileged Access Management

PAM solutions manage access to highly sensitive systems and credentials (root, DBA, Azure Owner, Active Directory admin).

```
PAM CAPABILITIES

JUST-IN-TIME ACCESS (JIT):
  Request: "I need admin access to production DB for 1 hour"
  Approval: manager + peer approves (optional, configurable)
  Grant: time-limited admin access for 1 hour
  Expire: access revoked automatically; session recorded
  Audit: full audit trail of who requested, who approved, what they did

CREDENTIAL VAULTING:
  Store privileged credentials (root passwords, service account passwords)
  Check out: request credentials → vault provides → time-limited
  Check in: credential rotated after return (optional auto-rotation)
  Break glass: emergency access with mandatory post-incident review
  Products: CyberArk, BeyondTrust, HashiCorp Vault, Azure Key Vault + PIM

SESSION RECORDING:
  Record all privileged sessions (SSH, RDP, DB)
  Searchable playback: index commands, SQL queries within session
  Tamper-evident: recording stored in vault, not on target machine
  Legal admissibility: chain of custody for forensics

AZURE PIM (Privileged Identity Management):
  JIT for Azure RBAC roles and Azure AD roles
  Eligible assignments: user can activate role when needed
  Active assignments: user always has role (use sparingly)
  Require approval + MFA + justification for activation
  Time-limited: 1h to 8h max activation
  Alert on permanent assignments, unused eligible assignments
```

## Service Account Hygiene and Workload Identity

Service accounts (credentials used by services, not humans) are a significant attack surface: they often have excessive privileges, long-lived credentials, and no MFA.

### The Problem with Long-Lived Secrets

```
TRADITIONAL SERVICE ACCOUNT:
  Config file: connectionString="Server=db;User=svc_payment;Password=abc123"

  Problems:
  - Secret is static: if leaked, compromised indefinitely until rotated
  - Secret is in config: may end up in source code, logs, dumps
  - Rotation is painful: coordinated deploy needed to change password
  - No MFA possible: machine-to-machine cannot do interactive MFA
  - Privilege audit: hard to track what the account actually accessed
```

### Workload Identity Federation

The modern solution: no long-lived secrets at all. Use ephemeral, auto-rotated tokens issued by the platform.

```
AZURE MANAGED IDENTITY:
  System-assigned: tied to resource lifecycle (VM, App Service, AKS pod)
    When resource is deleted → identity is deleted
  User-assigned: standalone identity, can be assigned to multiple resources
    Useful when multiple resources share the same permissions

HOW IT WORKS:
  Service running on Azure VM calls Azure Instance Metadata Service:
    GET http://169.254.169.254/metadata/identity/oauth2/token?resource=...
    → Returns short-lived access token (1 hour)
    → Auto-renewed before expiry
    → No credential stored anywhere (attacker cannot steal a static secret)

  Service then calls Azure API with the token:
    GET https://management.azure.com/...
    Authorization: Bearer {token}

KUBERNETES WORKLOAD IDENTITY (AKS):
  Pod is federated with an Azure AD service principal
  Pod gets projected service account token (Kubernetes-issued)
  Token exchanged for Azure AD token via OIDC federation
  → Pod has Azure identity without any secrets in the pod

GITHUB ACTIONS TO AZURE:
  Configure Azure AD app registration with GitHub OIDC federation
  GitHub Actions workflow gets OIDC token from GitHub
  Exchange for Azure AD access token
  → CD pipeline accesses Azure without storing client_secret
```

### Azure AD (Entra ID) Architecture

```
ENTRA ID CONCEPTS

TENANT: An organization's instance of Azure AD
  Identified by Tenant ID (GUID) or domain (contoso.onmicrosoft.com)
  All users, groups, applications, devices in one organization

DIRECTORY: The identity store within a tenant
  Objects: users, groups, service principals, managed identities, devices

SERVICE PRINCIPAL:
  The identity for an application within a tenant
  Created when app is registered (App Registration)
  Or when managed identity is created
  Has object ID, client ID, optional client secret or certificate

MANAGED IDENTITY:
  Type of service principal managed by Azure
  No credentials to manage; Azure handles token issuance
  Two types: system-assigned (per-resource) or user-assigned (shared)

APP REGISTRATION vs. SERVICE PRINCIPAL:
  App Registration: the definition (what is this app, what permissions does it need)
  Service Principal: the instance (this specific deployment of the app in this tenant)
  Multi-tenant apps: one App Registration → multiple Service Principals in multiple tenants
```

## Common Confusion Points

**"OAuth 2.0 is authentication"**
OAuth 2.0 is authorization — it grants access to resources. It does not verify who the user is. OIDC adds authentication on top of OAuth 2.0. When someone says "we use OAuth for login," they typically mean they use OAuth 2.0 + OIDC.

**"Access tokens are JWTs"**
Access tokens may or may not be JWTs depending on the authorization server. Azure AD v2 access tokens are JWTs. But the OAuth 2.0 spec does not require JWT — the format is opaque from the client's perspective. Only validate audience, expiry, and signature — do not assume internal JWT structure.

**"Service principal = service account"**
Close but not identical. A service account is a credential (username/password) for a service. A service principal is an identity in Azure AD. Service principals can use client secrets (like a password), certificates, or be managed identities (no credential). Prefer managed identities — they eliminate the "service account password" problem.

**"MFA is enough for privileged access"**
MFA prevents credential stuffing and phishing-based account takeover. It does not prevent an attacker who has already compromised the session (session token theft), or an insider threat. PAM (JIT + session recording) adds defense depth beyond MFA.

## Decision Cheat Sheet

| Scenario | Recommended Approach |
|----------|---------------------|
| Web app user login | Authorization Code + PKCE + OIDC |
| Mobile app user login | Authorization Code + PKCE + OIDC |
| Service calling another service | Managed Identity (Azure) or Client Credentials + certificate |
| CI/CD pipeline to Azure | GitHub OIDC Federation → Azure AD (no client_secret) |
| CLI tool user login | Device Flow |
| Existing enterprise SSO (SAML) | SAML 2.0 (legacy) or migrate to OIDC |
| Fine-grained document/resource sharing | ReBAC (SpiceDB, OpenFGA) |
| Production database access by engineer | PAM with JIT (Azure PIM) |
| Service needs SQL connection string | Key Vault reference in app config (not hardcoded) |
| Azure resource access from code | Managed identity (system-assigned for simple cases) |
